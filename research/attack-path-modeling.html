Attack Path Modeling in Segmented Enterprise Networks

Network segmentation reduces blast radius but rarely stops a determined attacker from reaching Tier Zero. This note walks through why — modeling segmentation not as walls but as a graph of identity, service, and trust edges — and what that means for where defenders should invest.

01 · The segmentation fallacy

Every enterprise network diagram looks segmented. VLANs, DMZs, restricted subnets, PAW enclaves, cloud VPC boundaries — the topology makes it look like the attacker has to break through a sequence of walls. In practice, attackers do not experience segmentation as walls at all. They experience it as a graph, and they look for the cheapest path through that graph.

The gap between "how the architecture looks" and "how the attacker moves" is where most breaches live. Every incident this author has reviewed over the past two years involved at least one edge the defenders either didn't know existed or had explicitly accepted as a risk and then forgotten about.

Thesis
Segmentation doesn't fail because the walls are weak. It fails because the graph has edges the defenders didn't draw. Modeling the attack surface as a graph, not as a topology, is the single highest-leverage change most defenders can make.

02 · Modeling segmentation as a graph

The useful abstraction: every principal in the environment — user, service account, host, cloud role, service identity — is a node. Every relationship that could be used to move from one to another is a directed edge.

Edges include the obvious ones (login, RDP, SSH) but also the subtle ones: constrained delegation, resource-based constrained delegation, shared credentials in CI secrets, ACLs on Group Policy Objects, AD object ownership, IAM trust relationships, service accounts with write access to config repos, and so on. Network topology is just one type of edge in this graph. Usually not even the most consequential type.

Figure 1 · Segmentation topology vs. attack graph
TOPOLOGY VIEW (what defenders see): [ Corp VLAN ] --firewall-- [ Server VLAN ] --firewall-- [ Tier 0 ] | | | workstations app servers DCs ATTACK GRAPH VIEW (what the attacker traverses): WS-042 ─login─▶ svc_build ─owns─▶ GPO ─applies─▶ DC01 │ │ ├──shared-secret──▶ Jenkins ─deploys──▶ AppSrv │ │ │ │ │ has-write │ │ ▼ │ └──session──▶ admin@tier1 ──SPN──▶ svc_sql ─RBCD─▶ DC02
The two views describe the same environment. The left view suggests three firewall hops between a workstation and a domain controller. The right view shows three edges — a shared secret, a GPO ACL, and a resource-based constrained delegation — that each bypass all three firewalls.

03 · Edge types attackers actually use

Across reviewed incidents, five edge categories account for the overwhelming majority of traversal:

3.1 Credential edges

Stored credentials on one host that authenticate to another — the classic. Appears in Windows credential manager, SSH keys, cloud credential files, CI/CD secrets, .git-credentials, browser-saved passwords, and service connection strings pulled into memory.

3.2 ACL edges

Write access to an object that controls another principal. GenericAll on a user, write on an OU's GPO, ownership of a group, WriteDACL on an object. These edges are essentially invisible to network-centric security monitoring.

3.3 Delegation edges

Kerberos constrained and resource-based constrained delegation. Both allow one identity to request tickets as another under specific conditions. RBCD in particular lets any attacker with WriteProperty on a computer object impersonate any user to that computer's services.

3.4 Supply-chain edges

One system's code or configuration is produced by another system. CI pipeline writes the deploy artifact. A config repo controls runtime behavior. An internal package registry is referenced by production. Compromising the producer compromises all consumers.

3.5 Trust-policy edges

AWS IAM trust policies, Azure managed identities, cross-tenant OAuth consents, SAML federation between tenants. Increasingly the common path in cloud-heavy environments — and the most commonly misconfigured.

04 · Five recurring traversal patterns

4.1 Jump-host bypass via delegation

The jump host enforces strong auth, session recording, and network ACLs. The service account the jump host runs under has constrained delegation to the target. The attacker never touches the jump host as a human — they abuse the service account's delegation rights from a compromised peer.

4.2 CI pipeline as universal backdoor

Production is segmented from corporate, but the CI system sits in corporate and pushes artifacts directly into production. Compromising a developer laptop leads to write access on the pipeline, which leads to arbitrary code execution in production — crossing every firewall in the path in a single step.

4.3 Shared identity across tiers

A single service account runs jobs in both a low-sensitivity tier and a high-sensitivity tier. Stealing its credential from the low tier grants a foothold in the high tier, regardless of network segmentation.

4.4 Configuration repo as tier-zero proxy

Infrastructure-as-code or config-as-code repositories effectively are tier zero, because they define what tier zero runs. A repo with weak access controls becomes the cheapest path to domain admin, bypassing every host-based control on the DCs.

4.5 AD ACL chains

A user has write access to Group A, which is a member of Group B, which has an ACL on a GPO, which applies to tier-zero machines. Each individual edge looks benign in isolation. The chain is catastrophic. These chains are what tools like BloodHound were built to find, and they're still wildly under-addressed in most environments.

Operational note
The five patterns above are not exotic. Every single one has appeared in multiple publicly-disclosed incidents. Defenders who model segmentation as a graph discover them in their own environments. Defenders who model segmentation as walls never see them until someone else walks through.

05 · Where defense has real leverage

Given the model, the highest-leverage defensive work is almost never "add another firewall." It is graph reduction: finding edges that shouldn't exist and removing them, and making the remaining edges more observable.

  1. Inventory identity edges explicitly. Service accounts, shared credentials, delegation relationships, IAM trust policies, OAuth consents. Most environments have never enumerated these.
  2. Eliminate tier-mixing service accounts. The same identity should not appear in both tier zero and tier one. Hard to achieve, cheap to audit.
  3. Treat CI pipelines and config repos as tier zero. Apply tier-zero controls: separate credentials, restricted approvers, strong auth, and audit.
  4. Run BloodHound (or equivalent) quarterly, not once. New edges appear continuously. The value is in trend, not snapshot.
  5. Monitor on the edges, not the walls. Alert on ACL changes, delegation changes, new IAM trust relationships, new Kerberos TGT patterns — these are much higher-signal than perimeter traffic anomalies.
