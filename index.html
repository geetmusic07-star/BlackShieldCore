Case Studies

Incident
Analysis

Post-mortem breakdowns of real security incidents. Root cause, exploitation path, and remediation — documented in full.

Forensic Timeline EVIDENCE SCAN
Financial Services

Bypassing 2FA via Race Condition

Analysis: Identified a severe multi-threading flaw in the OTP validation microservice of a major payment gateway. Simultaneous parallel requests exhausted and bypassed rate-limiting counters during the token lifecycle phase.

Remediation: Implemented strict database row-level locking and atomic Redis operations during token validation. Deployed canary monitoring across all OTP endpoints.

Vulnerability escalated through responsible disclosure. Patched globally within 72 hours.

Enterprise SaaS

LLM Prompt Injection Shield

Problem: Enterprise AI agents were vulnerable to adversarial prompts that could chain tool-use permissions into unauthorized data access, resulting in customer PII leakage.

Solution: Developed an intermediary neural filter trained on adversarial payloads. Deployed as an inline proxy with minimal inference overhead. Continuously improved via red-team feedback cycles.

Strong detection rate in testing. Low false positive rate across 18-day evaluation period.

Cloud / SaaS

API Rate Limit Bypass via Distributed Request Fragmentation

Context: A high-volume payment processing API enforced per-IP rate limiting on its authentication endpoint. By distributing requests across multiple edge nodes and fragmenting auth sequences, the limit was bypassed entirely without triggering any alert.

Fix: Replaced IP-based limiting with account-level and session-level tracking. Added velocity checks at the identity layer regardless of source IP. Retry fingerprinting added to flag high-frequency partial-auth patterns.

Disclosed through coordinated disclosure. Rate limiting architecture redesigned within 30 days.

Web Application

Stored XSS via Unsanitized Rich Text in Admin Portal

Issue: An enterprise SaaS platform's internal admin portal allowed rich text input in a user notes field. The field was sanitized client-side but the raw HTML was stored and rendered for admin users — creating a stored XSS path with admin session access.

Remediation: Server-side sanitization using DOMPurify-equivalent processing added before storage. Content Security Policy headers tightened to disallow inline script execution. All existing stored records were re-sanitized in a one-time migration.

Exploitable admin session hijacking path closed. No evidence of prior exploitation found in audit logs.

Infrastructure

Container Escape via Privileged Docker Run in CI/CD Pipeline

Discovery: A CI/CD pipeline ran Docker build steps with --privileged flag set by default. A compromised build script could mount the host filesystem, read secrets from other containers, and modify systemd unit files on the host.

Resolution: Privileged mode removed from all pipeline containers. Replaced with specific capability grants via --cap-add only where required. Build environments migrated to rootless Podman. Secrets scanning added to pipeline pre-commit hooks.

Complete host-escape path closed. CI/CD hardening guide published internally as part of the remediation.

Offensive / Defensive Tradeoffs in Security Lab Design

Designing a lab that's useful for both red and blue team work requires deliberate architectural decisions. Environments optimized for attack simulation tend to be too simplified for meaningful detection work, and vice versa. This note documents the tradeoffs encountered building the BSC home lab and the compromises that held up in practice.

  • Full AD replication introduces detection noise that masks real anomaly signals
  • Minimal viable enterprise simulation requires at least 3 network segments
  • Shared infrastructure between red and blue teams creates subtle attacker advantages in timing

OAuth 2.0 Implementation Flaws: A Pattern Analysis

OAuth 2.0 is well-specified but frequently misimplemented. This note catalogs recurring implementation patterns — state parameter omission, implicit flow misuse, PKCE bypass, and open redirect abuse — observed across real-world applications, with reproduction steps and detection guidance.

  • State parameter absent or static in ~40% of tested integrations
  • Redirect URI validation commonly implemented client-side only
  • Token leakage via Referer header in legacy fragment-based flows
  • PKCE frequently implemented but not enforced server-side

BlackShield Core is built as a structured research and learning environment covering the full breadth of modern cybersecurity — from offensive techniques and detection engineering to AI security and open-source intelligence.

Everything here is built to be useful, documented, and reproducible. The labs are grounded in real attack patterns. The research reflects genuine investigation, not manufactured content. The tools are built because they were needed, not because they look good on a portfolio.

The platform is actively expanding. Content is added as work progresses — not on a schedule, but when there's something worth documenting.

Build for understanding
Every lab, tool, and analysis exists to deepen technical understanding — not to generate content volume.
Document the reasoning
Results matter, but so does the methodology. How you got there is as important as what you found.
Stay adversarial
Defensive work that hasn't been tested against realistic attack techniques isn't reliable. Red team thinking informs everything here.
No overclaiming
Exact detection rates, guaranteed latencies, and zero false positives are marketing language. This platform avoids them deliberately.
Responsible disclosure
All vulnerability research follows coordinated disclosure practices. Nothing is published before affected parties have had time to respond.
Open by default
Lab configs, detection rules, and tool code are shared publicly where possible. Security improves when knowledge is accessible.
