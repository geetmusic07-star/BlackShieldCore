Supply Chain Compromise via npm Dependency Confusion

Case Study · Engagement: Q1 2025 (anonymized) · 15 min read · Published Apr 2026
Summary
Internal npm scoped package name was never registered on the public npm registry. An attacker (in this case a contracted red team) published a malicious package under the same name and higher version. The build system installed the public one, executing arbitrary code during npm install inside the production CI pipeline.
Outcome
Caught at detonation in CI via an outbound-connection rule to an unexpected domain. No production artifacts were deployed. Remediation completed within 72 hours. Systemic root cause (unregistered scope + no lockfile enforcement) addressed across all repos within two weeks.

Context

The client operated ~280 internal npm packages under a single organizational scope. Packages were hosted on a private Verdaccio registry with a fallback to the public npm registry for external dependencies. This fallback is standard — and is also the structural weakness that dependency confusion exploits.

Not all internal scoped packages had been registered on the public npm registry as placeholders. The assumption was that the private registry would always serve internal names first. In practice, the resolution behavior depends on lockfile handling, registry configuration, and npm client version — and in this engagement, the assumption did not hold.

Attack path

The engaged red team enumerated the client's public code surface and identified internal package names from leaked package.json files in public Stack Overflow posts, open-source SDK repos, and the client's own npm organization page (which listed every scoped package regardless of registry). From that list they selected a package with:

  • No public-registry placeholder
  • Active usage across multiple internal repos (inferred from version history)
  • No pinned lockfile entries in at least one CI pipeline they could see in public CI logs

They published a malicious version to public npm at a version number higher than anything the client used internally. The package contained a postinstall hook that attempted a DNS lookup to an attacker-controlled domain, tagged with the build host's identity.

Timeline
T+0hRed team publishes malicious package to public npm registry under scope @client-scope/internal-util at version 9.9.9.
T+3hScheduled nightly CI run on the main product repo pulls the public version instead of the internal one due to lockfile drift.
T+3h02mPostinstall hook fires. DNS lookup to attacker-controlled domain is logged and flagged by the egress firewall.
T+3h07mOn-call engineer receives egress alert. CI job is killed. Image is not pushed.
T+5hIncident response initiated. Red team notifies client. Engagement exits covert phase.
T+72hAll ~280 internal scoped names registered as public placeholders. Lockfile enforcement enabled in CI. Build host network posture tightened.

What went wrong

FINDING 01
Internal scope names were publicly enumerable
The organization's npm page listed every scoped package, including ones with no public placeholders. Combined with leaked package.json snippets, this gave an attacker a complete target list without needing any access to the client's infrastructure. The attacker's intel-gathering phase was trivially shortened.
FINDING 02
CI ran npm install without strict lockfile enforcement
The nightly build used npm install rather than npm ci, and the registry config was configured to fall through to the public registry on private registry miss. This meant a newer version on the public registry would be selected even when the lockfile had an entry pointing at the private one. The fallback behavior was the actual vulnerability.

What went right

The only reason this didn't become a production incident is that the egress firewall enforced a default-deny outbound policy for build hosts, with explicit allowlists for the public npm registry, the internal registry, and a handful of known external dependencies. The DNS lookup attempt to the attacker's domain was blocked and logged. The alert routed to an on-call rotation that actually responded.

Two separate controls were required for the catch: the egress policy existed, and the alerting pipeline on top of it was real. Many organizations have the first and not the second.

Takeaway
Dependency confusion is not a novel vulnerability. It is a well-understood structural flaw that still catches organizations because the mitigations are operational (register placeholder names, enforce lockfiles, restrict egress from build hosts) rather than a single technical fix. The mitigations are also cheap. Organizations that haven't done them usually haven't done them because nobody owns the end-to-end path — not because the work is hard.

Recommendations applied

  • Register every internal scoped name on the public registry as a placeholder with a stub package
  • Replace npm install with npm ci in all CI pipelines
  • Configure the private registry with strict scope-based routing (no public fallback for the internal scope)
  • Default-deny egress on build hosts with an explicit allowlist of registries
  • Quarterly audit of scoped package registration coverage
